# AI Front Desk

A simple, self-hostable **client dashboard for [Assistable.ai](https://assistable.ai)**.
Give each client a clean login to see what their AI voice agent handled — calls,
recordings, transcripts, sentiment, and chat volume — instead of handing them the
Assistable UI or a GoHighLevel iframe.

Built for people already using the Assistable **API / CLI / MCP**: point it at your
API key + a subaccount, deploy it, and send each client a URL and a login. It reads
Assistable through a server-side proxy, so your API key never touches the browser and
each client only ever sees their own, read-only data.

| | |
|---|---|
| **Overview** | Calls handled, completion rate, avg duration, sentiment, daily volume — per time window (7 / 30 / 60 / 90 days). |
| **Calls** | Searchable, filterable list with plain-English summaries and masked caller info. |
| **Call detail** | Summary → recording player → clean transcript → sentiment. |
| **Multi-tenant** | One deploy serves many clients; each is pinned to its own Assistable subaccount. |
| **Admin view** | An agency admin can switch between all clients from the header. |
| **Zero-config demo** | Ships with a built-in demo login backed by fully synthetic data. |

## Why

If you run AI voice agents for clients on Assistable, the reporting options are the
Assistable dashboard or an embedded GHL view — both are more than most clients want and
harder to white-label. This is the opposite: one URL, one login, a read-only view of just
their numbers. Self-hosted and yours to brand.

## How it works

Everything runs server-side in a small Next.js app. The browser never sees your API key
or another client's data:

```
Browser ──▶ Next.js (this app)
              • signed, httpOnly session cookie → which client you are
              • BFF proxy holds ASSISTABLE_API_KEY, pins X-Subaccount-Id to your session
              • allowlist-maps every response to a client-safe DTO (src/lib/types.ts)
                — no cost/revenue, no internal IDs, no raw PII
            └─▶ api.assistable.ai/v3
```

Recordings stream through an authenticated proxy route, so the raw audio URL is never
exposed either. Passwords are stored only as scrypt hashes.

## Requirements

- [Bun](https://bun.sh) (or Node.js 20+)
- An [Assistable.ai](https://assistable.ai) account with an **API key** and at least one
  **subaccount ID**
- A [Vercel](https://vercel.com) account (or any Node host) to deploy

## Quickstart

```sh
# 1. Clone + install
git clone https://github.com/angelo-swe/ai-front-desk-oss.git
cd ai-front-desk-oss
bun install

# 2. Configure
cp .env.example .env.local

# 3. Generate a session secret and a tenant password hash
openssl rand -hex 32                  # → AFD_SESSION_SECRET
bun scripts/hash-password.ts hunter2  # → passwordHash for a tenant/admin

# 4. Run
bun run dev                           # http://localhost:3000
```

Open [localhost:3000](http://localhost:3000) and log in with **`demo@frontdesk.app`** /
**`demo`** to browse the synthetic sample data — no Assistable account needed. To wire a
real client, fill in `ASSISTABLE_API_KEY` and add a tenant to `AFD_TENANTS` (below).

## Configuration

All configuration is environment variables (see [`.env.example`](.env.example)):

| Variable | Required | What it is |
|---|---|---|
| `AFD_SESSION_SECRET` | yes | Secret that signs the session cookie. `openssl rand -hex 32`. |
| `ASSISTABLE_API_KEY` | for live data | Your Assistable API key. **Server-only** — never sent to the browser. |
| `AFD_TENANTS` | for live data | JSON array of clients: `slug`, `email`, `passwordHash`, `name`, `assistantName`, `timezone`, `subaccountId`, optional `slackChannelId`. |
| `AFD_ADMINS` | optional | JSON array of agency admins who can switch between all clients. |
| `NEXT_PUBLIC_AGENCY_NAME` | optional | Your brand, shown across the UI. |
| `AGENCY_CONTACT_EMAIL` | optional | Target for the Settings "Request a change" button (hidden if unset). |
| `KV_REST_API_*` / `UPSTASH_REDIS_REST_*` | optional | A Vercel KV / Upstash store. When set, admins can edit each client's Slack channel ID in-app instead of only via `AFD_TENANTS`. |

Passwords are never stored in plaintext — `AFD_TENANTS` / `AFD_ADMINS` hold scrypt hashes
generated by `bun scripts/hash-password.ts <password>`.

## Deploy

[![Deploy with Vercel](https://vercel.com/button)](https://vercel.com/new/clone?repository-url=https://github.com/angelo-swe/ai-front-desk-oss)

Set the same environment variables in your host. On Vercel, add `ASSISTABLE_API_KEY` (and
the other secrets) with the **Sensitive** flag on.

- `/overview` — hero KPI + cards + daily-volume chart + sentiment + recent calls

## Multi-tenant & admin

Each entry in `AFD_TENANTS` is a client, pinned to its own Assistable `subaccountId`; a
client login only ever sees that subaccount. Accounts in `AFD_ADMINS` can switch between
all clients from the header. The built-in `demo` tenant serves the static synthetic
snapshot and is excluded from the switcher.

## Tech stack

Next.js 16 (App Router) · React 19 · TypeScript · Tailwind v4 (CSS-only theme, no config
file) · Motion v12 · lucide-react. Light + dark mode via a tiny custom provider (no
dependency). Package manager: Bun.

## Caveats

- **Unofficial.** Not affiliated with, endorsed by, or supported by Assistable.ai. It just
  uses their public API.
- **Your API key is powerful.** A single Assistable key is typically read-write across all
  your subaccounts. This app only ever reads, and keeps the key server-side — but treat it
  like any production secret, and use a read-only key if your plan offers one.
- **Metrics are real only.** The dashboard shows what Assistable reports (calls, duration,
  sentiment, chats). It intentionally does not invent revenue/ROI numbers.
